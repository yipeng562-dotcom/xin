# 踩坑记录

更新时间：2026-07-07

## 记录规则

- 每次我给出错误方向、过早假设、遗漏用户已确认规则、代码结构不够抽象，都记录在这里。
- 记录格式：时间 / 场景 / 我踩的坑 / 正确做法 / 后续规则。
- 如果某条坑会长期影响代码风格，同时同步精炼到 `D:\xin\AGENTS.md`。

## 2026-07-07 职业技能冷却变量设计

- 场景：讨论素问战斗技能库，技能1无冷却，技能2冷却6秒。
- 我踩的坑：一开始想让用户加 `素问技能2上次时间` 这种单职业单技能变量。
- 问题：以后每个职业、每个技能都要加变量，会越来越乱。
- 正确做法：统一用二维冷却表，例如 `技能上次时间 [账号序号] [技能编号]`，所有职业共用一套冷却记录。
- 后续规则：以后写职业技能库，先设计统一技能配置和统一冷却表，再写职业分支。

## 2026-07-07 单独拆素问技能子程序

- 场景：战斗释放技能代码。
- 我踩的坑：把 `释放素问技能` 单独拆出来，又在 `释放职业技能` 里判断职业。
- 问题：职业在启动/登录配置阶段已经确定，释放技能这里再判断职业是重复逻辑；单独拆素问也会导致以后每个职业都多一份类似代码。
- 正确做法：使用统一的“释放当前技能”入口，按当前已配置好的技能表和冷却表释放技能。
- 后续规则：不要再把某个职业单独拎出来写一套释放子程序；当前手动测试直接调用当前技能释放入口。

## 2026-07-07 技能教学后续状态写死

- 场景：技能教学里有飘絮，也有燕回风。
- 我踩的坑：一开始把技能教学都当成同一种流程，处理完就回准备任务。
- 问题：飘絮教学后可以回准备任务，但燕回风是在战斗链路中，要继续战斗中循环按 A 或继续观察战斗。
- 正确做法：技能教学子程序用 ret 分流，`ret=飘絮` 和 `ret=燕回风` 分别走不同后续状态。
- 后续规则：同一范围同一颜色的教学提示，优先一条 `FindStrFast` 多关键词，用 ret 判断动作。

## 2026-07-07 必看剧情结束后的去向

- 场景：出现“显示界面”的必看剧情，看完后接 F 对话、技能教学或战斗提示。
- 我踩的坑：一开始想在必看剧情结束后直接调用 `处理对话F`。
- 问题：剧情结束后不一定只出现 F，也可能出现技能教学、战斗目标、交互提示。
- 正确做法：必看剧情只负责等待“显示界面”连续消失，然后返回给任务状态机，由“观察任务结果”统一分诊。
- 后续规则：单个处理函数只做自己的职责，不要替后续流程猜下一步。

## 2026-07-07 没有优先使用 ret 分流

- 场景：处理“服下”、技能教学、战斗目标等识别。
- 我踩的坑：一开始给单个关键词写独立子程序或多段重复找字。
- 问题：代码量大，后续每补一个关键词都要复制一段，容易漏调试输出和状态切换。
- 正确做法：同范围、同颜色、同类动作，写成一条多关键词识别，用 ret 分流。
- 后续规则：除非范围、颜色、动作差异很大，否则优先 ret 分流。

## 2026-07-07 ret 分流格式没有完全按用户示例

- 场景：用户给了梦幻脚本示例，要求学习 ret 分流格式，并明确不要进程ID。
- 我踩的坑：之前只理解成“用 ret”，但没有完整吸收格式：先 `x＞0` 再 `.判断开始`，同动作 ret 合并，特殊 ret 单独分支，ret2 二次确认，短循环找到后跳出。
- 问题：如果只会零散使用 ret，还是会写成多个 FindStrFast 或分支混乱。
- 正确做法：按示例结构写：`FindStrFast -> 调试输出(ret,x,y) -> x＞0 -> 计次清零 -> 判断开始(ret...) -> 默认兜底；否则计次兜底`。
- 后续规则：本项目点击不带进程ID，统一使用 `左键单击(大漠,x,y)` 等现有封装。

## 2026-07-07 无必要引入 ret2

- 场景：整理 `处理主线准备任务`。
- 我踩的坑：在用户不需要二次确认时引入 `ret2`。
- 问题：代码能用但和用户要的简洁 ret 分流格式不一致。
- 正确做法：默认只用一个 `ret`；示例里的 `ret2` 只是因为前面 ret 已经占用。
- 后续规则：以后不要主动引入 `ret2`；只有同一作用域必须同时保留两个识别结果时才允许使用。

## 2026-07-07 改代码说明不够细

- 场景：用户问“怎么改”“放哪里”“删哪里”。
- 我踩的坑：有时只给代码，没有明确说明替换哪个子程序、删除哪一段。
- 问题：用户需要自己猜，很容易贴错位置或保留冲突代码。
- 正确做法：每次改动必须说明“替换整个子程序 / 只加在某段前面 / 删除某行调用 / 新增子程序”。
- 后续规则：除非用户只要代码，否则代码前后要简短说明改动位置。

## 2026-07-07 重复漏说代码放哪里

- 场景：给战斗释放技能代码时，用户需要知道程序集变量放哪里、哪些是新建子程序、哪里要替换。
- 我踩的坑：已经在 MD 里写过“要说清楚放哪里”，但仍然先发代码，漏了放置说明。
- 问题：这是重复踩坑，会让用户无法直接粘贴和测试。
- 正确做法：以后每次给代码，正文第一段必须写清楚：`新增 / 替换 / 插入 / 删除` 四类动作。
- 后续规则：如果代码超过一个子程序，必须先列清单，再发代码。

## 2026-07-07 靠记忆说手动任务里有初始化位置

- 场景：用户贴出 `_按钮_手动任务_被单击`，里面并没有我之前说的那段初始化位置。
- 我踩的坑：我没有先确认当前代码，就按记忆说“在任务状态机前加初始化”。
- 问题：当前代码是启动线程进入任务状态机，不是直接调用；靠记忆会把用户绕糊涂。
- 正确做法：忘记当前结构时，必须让用户贴当前子程序，或者先读取附件，再给修改方案。
- 后续规则：涉及“插入到哪里”的问题，必须基于用户当前代码原文，不许靠上一轮记忆猜。

## 2026-07-07 切画面后只检测一次

- 场景：必看剧情“显示界面”消失后，后面会接技能教学/对话/F/战斗提示。
- 我踩的坑：一开始只在剧情结束后调用一次 `处理技能教学(i)`。
- 问题：游戏切画面有刷新延迟，只检测一次很容易刚好没刷出来，导致误判为没有后续状态。
- 正确做法：切画面后的状态默认连续检测 2 次，每次加调试输出。
- 后续规则：所有转场/剧情结束/弹窗关闭/F后刷新/技能教学刷新，都默认检测 2 次，不再只查一次；如用户单独指定次数，以用户指定为准。

## 2026-07-07 切画面短循环命中后没有统一跳出

- 场景：切画面后连续检测技能教学/对话/F/战斗提示。
- 我踩的坑：一开始倾向于在循环内部找到就直接 `返回真`，没有强调“执行完跳出循环，循环后统一处理状态”。
- 问题：循环内部到处返回，状态切换容易分散；找到后如果不跳出，也可能继续卡在循环里重复找。
- 正确做法：短循环内找到并执行后，设置 `是否已处理＝真`，然后 `跳出循环()`；循环结束后统一切状态或返回。
- 后续规则：切画面短循环优先用“找到 -> 执行 -> 标记 -> 跳出 -> 循环后统一收口”的写法。

## 2026-07-07 错把必看剧情加最大检测次数

- 场景：不可跳过剧情/必看剧情通过“显示界面”判断是否还在播放。
- 我踩的坑：建议加最大检测次数，达到次数后强制跳出。
- 问题：不可跳过剧情必须真的看完，不能因为达到次数就跳出，否则会提前离开剧情等待，后续状态会乱。
- 正确做法：只允许“显示界面”连续多次找不到后跳出；如果还一直识别到显示界面，就继续等待。
- 后续规则：不可跳过剧情循环不加最大次数强制跳出，只靠界面连续消失作为正常退出条件。

## 2026-07-07 混淆普通短循环和必看剧情长等待

- 场景：补回 `处理主线观察任务结果(i)` 时，用户确认先使用快照重建版。
- 我踩的坑：容易把普通状态分诊循环写成一直等待，或者把必看剧情写成固定次数退出。
- 问题：普通状态分诊如果不退出，会卡住后续状态；必看剧情如果固定次数退出，会提前离开剧情。
- 正确做法：普通短循环找到目标就标记并跳出，找满用户确认次数仍未找到也跳出；不可跳过剧情/必看剧情只允许等界面连续消失后退出。
- 后续规则：`观察任务结果` 这类分诊逻辑默认是 2 次短循环；只有“显示界面”这类不可跳过剧情才用连续消失作为唯一正常退出条件。

## 2026-07-07 对话F误写成只按一次

- 场景：`处理对话F(i)` 处理剧情对话里的 `F继续.bmp`。
- 我踩的坑：把对话F写成找到后只按一次，并且用了 3 次未找到和低于 1000 的等待。
- 问题：对话F会循环出现，直到没有 F 才算结束；只按一次会提前切回准备任务，导致右侧任务还没刷新。
- 正确做法：首次找到 F 后进入持续出现型循环；每次找到 F 就按 F，单次等待最少 `1000`，并重置未找到次数；连续 2 次找不到 F 后再切回 `准备任务`。
- 后续规则：对话F这类持续出现型循环不是固定总次数 2 次，而是连续 2 次找不到才结束；延时默认不低于 `1000`，除非用户明确给出其它数值。
